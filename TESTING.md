## 1. Как поднять стек

### 1.1 Бэкенд (порт 8777)

Из корня репозитория:

```bash
python3 server.py > server.out 2>&1
```

- Слушает `127.0.0.1:8777` (`ThreadingHTTPServer`, `server.py:10700`, `PORT = 8777` на `server.py:51`).
- **Биндится медленно (~10 c)** — поднимается FTS-индекс/каталог в фоне. Не дёргай curl сразу: `sleep 10`, потом проверяй.
- При старте печатает баннер (видно в `server.out`):
  ```
  ── Mostik AI · http://127.0.0.1:8777
  ── ключ venice: …  ✓/✗ нет
  ── моделей в каталоге: N · данные: …
  ```
- `server.py` сам в файл НЕ пишет — поэтому редиректим вывод в `server.out` (`> server.out 2>&1`). Это же делает координаторский рестарт.

**Чистый рестарт** (если порт занят / завис):
```bash
for p in $(lsof -ti:8777); do kill -9 $p; done; pkill -9 -f "python3 server.py"; sleep 2
python3 server.py > server.out 2>&1   # запускай в фоне (run_in_background), НЕ через & / disown
sleep 10
curl -s "http://127.0.0.1:8777/api/init?user=default" | head -c 200
```

**Переменные окружения бэкенда** (сверено по `grep os.environ` в `*.py`):

| Переменная | Где | Дефолт | Что делает |
|---|---|---|---|
| `TAIGA_LOG` | `server.py:2106` | `1` (ВКЛ) | Лог запросов в stdout (→ `server.out`). `TAIGA_LOG=0` — выключить лог запросов. |
| `MOSTIK_SLEEPTIME` | `scheduler.py:52` | пусто (ВЫКЛ) | Sleep-time обслуживание (фоновая консолидация памяти и т.п.) тикает ТОЛЬКО при `1/true/yes/on`. По умолчанию выключено — это нормально, не баг. |
| `VENICE_API_KEY` | `adapter_example.py` | — | Подхватывается из env, если файла ключа нет (вспомогательный путь, не основной — ключи лежат в файлах провайдеров). |

> `TAIGA_BASE` / `TAIGA_OWNER` / `TAIGA_SLOW` — это переменные **тест-сьюта** (`tests/test_endpoints.py`), а НЕ сервера. См. раздел 2.

### 1.2 Фронтенд (порт 3000)

Из `taiga-web/`:

```bash
# dev (с hot-reload, Turbopack)
npm run dev          # = next dev → http://localhost:3000  (приложение: http://localhost:3000/app)

# prod-сборка + запуск
npm run build        # = next build
npm run start        # = next start (порт 3000)
```

(сверено по `taiga-web/package.json` scripts и `taiga-web/dev.out`: «▲ Next.js 16.2.9 (Turbopack) · Local: http://localhost:3000»).

**Главный экран приложения — `/app`** (не `/`; `/` — лендинг). В dev-логе видно `GET /app 200`.

**Прокси на бэкенд.** Каждый `fetch("/api/X")` фронта идёт в `taiga-web/src/app/api/X/route.ts` (Next-прокси) → бэкенд. Адрес бэкенда:

```ts
const BACKEND = process.env.TAIGA_BACKEND ?? "http://127.0.0.1:8777";
```

(сверено: одинаковая строка во всех `src/app/api/*/route.ts`, напр. `src/app/api/init/route.ts:6`, `chat/route.ts:8`).
Значит для локального теста **достаточно поднять бэкенд на 8777** — фронт сам найдёт. Если бэкенд на другом порту/хосте — задай `TAIGA_BACKEND` перед `npm run dev`.

> Если в dev-логе мелькает `Blocked cross-origin request … /_next/webpack-hmr from "127.0.0.1"` — это безобидное предупреждение HMR (заходи через `http://localhost:3000`, не `127.0.0.1`). На функциональность не влияет.

## 2. Авто-тесты

### 2.1 Бэкенд: endpoint-смоук (16 тестов, нужен живой бэкенд)

```bash
# бэкенд ДОЛЖЕН быть запущен на 8777
python3 -m unittest tests.test_endpoints -v        # из корня репо
# или:  tests/run.sh
# или:  python3 tests/test_endpoints.py
```

- **16 тест-методов** (сверено: `grep -c 'def test_' tests/test_endpoints.py` = 16). Stdlib-only (`unittest`+`urllib`, без pip).
- Бьёт по **живому** бэкенду: реальные коды, JSON-ключи, порядок SSE-событий.
- Полный прогон **~1–2 минуты** (часть тестов реально зовёт модели).
- **Умные пропуски, а не падения:**
  - бэкенд недоступен → весь класс **SKIP** с понятным сообщением;
  - у провайдера нет ключа/баланса → конкретный тест **SKIP** (ловит маркеры «баланс»/«нет ключа»/«квота»/SSE `error`/JSON `402`);
  - структурные проверки (коды, форма JSON, owner-гейты, валидация входа) — **всегда жёсткие**, от баланса не зависят.

Env-переменные сьюта (`tests/test_endpoints.py:34-37`):

| Var | Дефолт | Смысл |
|---|---|---|
| `TAIGA_BASE` | `http://127.0.0.1:8777` | база бэкенда |
| `TAIGA_OWNER` | `default` | id владельца (он же owner per `server.is_owner`) |
| `TAIGA_SLOW` | `120` | таймаут (c) на ручки, зовущие модели |

Что покрывают (из `tests/README.md` + кода):

- **TestInit** — `GET /api/init` 200 + ключи `users/models/full/system/keys/balance/balances/billing/byok/settings/memory`; `billing.owner === true` для владельца.
- **TestChat** — SSE `meta → delta → done` по порядку; пустое сообщение → JSON `400` ДО стрима/биллинга.
- **TestCouncil** — `council:true` шлёт `council_plan` (со списком `members`) до обращения к провайдеру, затем `council_step/meta/done`.
- **TestSearchChats** — `POST /api/search_chats` → `{results,count,q}`, `count === len(results)`.
- **TestWorkflow** — `GET /api/workflow` отдаёт шаблоны (есть встроенный `rewrite-polish`); прогон шаблона → `{ok,steps,result}`.
- **TestSelftest** — `POST /api/selftest` (owner) → `{ok,passed,failed,total_ms,checks[]}`, `passed+failed === len(checks)`.
- **TestRag** — `rag_ingest` → `rag_query` возвращает hit с привязкой к doc; чистит за собой через `rag_delete`. SKIP если эмбеддинг-провайдер недоступен (`502`).
- **TestOrchestrate** — `POST /api/orchestrate` → `{plan,results,final,steps}` с непустым `final`; пустая задача → `400`.
- **TestOwnerGates** — `selftest` / `memory_consolidate` / `run` для НЕ-владельца (`stranger123`) → `403`.
- **TestInputValidation** — >400 сообщений → `400`; ~4.1 МБ текста → `400` (потолки `SEC_MAX_MESSAGES=400`, `SEC_MAX_TOTAL_CHARS=4_000_000`).

## 3. Владелец vs обычный пользователь

- **Владелец — это user `default`** (`server.is_owner`, `server.py:5079`: owner = `uid == "default"` или флаг `owner` у юзера).
- **Owner-only фичи** (обычному юзеру — `403` / скрыты):
  - `/api/selftest` (она же `/sprint`) — один из чеков зовёт платную модель, поэтому только владелец;
  - `/api/memory_consolidate` (sleep-time консолидация памяти);
  - `/api/run` (код-интерпретатор — анти-RCE гейт);
  - редактор identity / персоны (`/api/identity`);
  - панель диагностики и sub-meters (метры подписок/расхода);
  - бесплатные модели по подписке владельца (`OWNER_SUB_MODEL = ng:deepseek-ai/deepseek-v3.2`), плюс владелец освобождён от IP-rate-limit.
- **Как проверить как НЕ-владелец:** дай любой `user`-id кроме `default` (в тестах — `stranger123`). Через UI это смена id пользователя; через API — `{"user":"stranger123", …}`. Ожидание: owner-ручки → `403`, платные модели без баланса авто-скрываются, топап от не-владельца → `503` (см. `night-smoke.sh`).

## 5. Встроенный само-тест

Два пути, одна логика (owner-only):

1. **`/sprint`** в чате — перехватывается на клиенте и POST-ит `/api/selftest`, рендерит ✅/❌ по чекам.
2. **`POST /api/selftest {"user":"default"}`** напрямую → JSON `{ok, passed, failed, total_ms, checks:[{name, ok, ms, detail}]}`.

```bash
curl -s -X POST http://127.0.0.1:8777/api/selftest \
  -H 'content-type: application/json' -d '{"user":"default"}' | python3 -m json.tool
```

**6 подсистемных чеков** (сверено по `server.py:8437` `api_selftest`):
1. **Каталог моделей** — витрина + полный список + RICH грузятся.
2. **Биллинг и баланс** — тарификация и баланс читаются.
3. **База знаний (RAG)** — список доков доступен (без эмбеддингов, дёшево).
4. **Супер-поиск** — есть движки + хотя бы один grounding-ключ.
5. **Планировщик агентов** — раннер подключён, список джобов читается.
6. **Путь вызова модели** — самый дешёвый пинг (`max_tokens=1`); если ключа нет — это **пропуск (зелёный)**, а не провал.

Не-владелец → `403`. **Панель диагностики** в Настройках даёт то же самое в UI плюс пинг `/api/init` и кошельки.

## 7. Траблшутинг

| Симптом | Причина / что делать |
|---|---|
| curl/фронт сразу после старта бэкенда даёт connection refused | **Медленный бинд (~10 c)** — поднимается FTS/каталог в фоне. `sleep 10`, потом проверяй. |
| `401`/ошибка от провайдера в чате | Нет/протух ключ или нет баланса у этого провайдера. Проверь баннер ключей в `server.out` (`── ключ …: ✓/✗`) и панель диагностики. Тест-сьют такие случаи **пропускает**, не валит. |
| Платная модель пропала из каталога | **Авто-скрытие при нулевом балансе** — by design. Пополни баланс провайдера или выбери модель доступного провайдера. |
| Тест-класс целиком SKIPPED | Бэкенд недоступен на `TAIGA_BASE`. Подними `server.py` на 8777 и повтори. |
| Отдельный тест SKIPPED («баланс/нет ключа/квота») | Провайдер пуст — это не провал. Структурные тесты при этом всё равно прошли. |
| `Blocked cross-origin … webpack-hmr from 127.0.0.1` в dev-логе | Безобидное HMR-предупреждение. Заходи через `http://localhost:3000`, не `127.0.0.1`. |
| Фронт стучится не туда | Прокси берёт `process.env.TAIGA_BACKEND ?? http://127.0.0.1:8777`. Для нестандартного бэкенда задай `TAIGA_BACKEND` перед `npm run dev`. |
| Sleep-time память «не работает» | Она **выключена по умолчанию** (`MOSTIK_SLEEPTIME` не задан). Это норма. Включается `MOSTIK_SLEEPTIME=1`. |

### Как читать `server.out`

Сервер пишет по строке на запрос (вкл. при `TAIGA_LOG=1`, дефолт):

```
req method=GET  path=/api/init  status=200  ms=4370  uid=default
req method=POST path=/api/chat  status=400  ms=8465
req method=POST path=/api/selftest status=403 ms=0
req method=POST path=/api/chat  status=400  ms=0  err_type=JSONDecodeError err="…"
```

Читай так: `status=` — HTTP-код, `ms=` — время ответа (большие `ms` на `/api/chat`/`orchestrate` — это вызов модели, норма), `uid=` — кто звал, `err_type/err=` — текст исключения если было. `status=403` на owner-ручках от не-владельца — это правильное поведение, не баг.

Хвостить лог:
```bash
tail -f server.out
```
