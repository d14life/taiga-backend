# Тайга — план масштабирования до 10k одновременных heavy-coding юзеров

Честно: текущая архитектура (один stdlib-сервер + один SQLite + локальный MLX на Mac) — отличный
MVP/одна-нода, но 10k одновременных = ДРУГАЯ лига инфры. Ниже — конкретные шаги по приоритету.
Часть кода уже подстелена (#6 RAG-строки). Что требует денег/серверов — помечено 💳 (твои руки).

---

## Фаза 0 — что УЖЕ готово как фундамент (сделано в коде)
- **RAG по-кусочным строкам + индекс по uid** (`rag_chunks`) — прямой мост к pgvector (та же схема: uid, vec, text, ws). Миграция на Postgres = перелить строки, не переписывать вызывающих.
- **Гибрид-поиск + RRF + LLM-реранк** — алгоритм ретрива готов, переносится на pgvector как есть.
- **Авто-здоровье моделей** (phantom + recently-empty + fallback) — переживёт смену БД/нод (in-memory + per-model).
- **Per-user изоляция** памяти/RAG/файлов — мульти-тенантность уже в дизайне.

## Фаза 1 — БД: SQLite → Postgres + pgvector  💳(managed Postgres ~$20-50/мес)
Зачем: SQLite = ОДИН писатель. 10k конкурентных записей (чаты/память) он не вывезет.
Шаги:
1. Поднять managed Postgres (Supabase/Neon/RDS) + расширение `pgvector`.
2. Слой БД: заменить `_db()` (sqlite3) на пул psycopg (connection pool). Таблицы 1:1 переносятся.
3. `rag_chunks.vec` → тип `vector(1536)` (text-embedding-3-small = 1536 dim) + **HNSW-индекс**:
   `CREATE INDEX ON rag_chunks USING hnsw (vec vector_cosine_ops);`
4. Поиск: `WHERE uid=$1 ORDER BY vec <=> $2 LIMIT k` — предфильтр по uid (индекс) + ANN (HNSW). О(log n), тянет десятки млн векторов.
5. Гибрид нативно: pgvector (смысл) + `tsvector`/`ts_rank` (ключевые слова) в одном запросе.
Риск: средний. Делается за ~2-4 дня. Данные мигрируются скриптом (SQLite → PG dump).

## Фаза 2 — Сервер: один процесс → много нод за балансировщиком  💳(VPS/контейнеры)
Зачем: stdlib `ThreadingHTTPServer` = один процесс, GIL душит на 10k.
Шаги:
1. Обернуть в ASGI/WSGI (или переписать ручки под FastAPI) — мало кода, ручки те же.
2. Запуск через gunicorn/uvicorn с N воркер-процессов на ноду.
3. N нод за nginx/Caddy балансировщиком. Состояние — в Postgres (не в памяти процесса).
4. Вынести in-memory стейт (phantom, _AUX_EMPTY, agent_permit/answer) в Redis (общий между нодами).  💳(Redis ~$10/мес)
Риск: средний-высокий (рефактор стейта). ~1 неделя.

## Фаза 3 — Инференс  💳(ГЛАВНЫЕ деньги)
Это узкое место: 10k кодят = огромный поток генерации.
Вариант A (рекомендую старт): **остаёмся на API-провайдерах** (NanoGPT/Venice/др.) — у них флот, платишь за токен, наценка покрывает. Масштаб — их забота.
Вариант B (полный self-host): **GPU-флот** (десятки-сотни H100) — только если экономика API не сходится на объёме. Это серьёзные деньги + DevOps.
- **1 арендованный GPU** (A100/H100, ~$1-2/час): НЕ для главного инференса 10k, а под ИНФРУ →

## Фаза 4 — 1 GPU под инфру-модели (лучший ROI на старте)  💳(~$700-1500/мес за 1 GPU)
Одна арендованная карта обслуживает ВСЮ служебку для 10k (короткие вызовы, тысячи/сек):
- **Эмбеддер** (bge/e5) — свои эмбеддинги, не платить провайдеру за каждый чанк.
- **Reranker** (cross-encoder bge-reranker) — заменить LLM-as-judge → быстрее+бесплатно. Повесить `/rerank` рядом с `/compress` на :8791-аналоге.
- **CLaRa-компрессор** (текст) — уже есть локально, переносится на GPU-ноду.
- **Маленькая LLM (7-8B)** на дешёвую служебку (followups/plan/style/память + замена нестабильной gemma) — один GPU тянет всё.
Риск: низкий. Окупается сразу (снимает плату провайдеру за служебку + ускоряет RAG).

## Фаза 5 — Латент-CLaRa (R&D, ПОСЛЕ GPU)
Только при своём инференсе (open-weights + латент-инъекция в forward-pass — через API нельзя).
- Нужна open-модель + кастомный компрессор контекста в латент.
- Выигрыш: окно/токены (4-8× контекста), цена. НЕ галлюцинации (их режут ретрив+grounding+верификатор — уже сделано).
- Это исследовательский проект, не неделя. Делать, когда инфра (фаза 3-4) стоит.

## Рекомендованный порядок (по ROI/риску)
```
СЕЙЧАС бесплатно:   ничего срочно — гибрид-RAG + строки + авто-здоровье уже в проде
ПЕРВЫЕ деньги:      Фаза 4 (1 GPU под инфру) — дёшево, окупается, ускоряет RAG/служебку
При реальном росте: Фаза 1 (Postgres+pgvector) → Фаза 2 (много нод+Redis)
Инференс:           Фаза 3A (API) пока экономика сходится; флот/self-host — потом
R&D:                Фаза 5 (латент-CLaRa) — последним, на своём GPU
```
**Главное:** до ~сотен одновременных юзеров можно НЕ трогать инфру (текущая нода тянет). 10k — это
фазы 1-2 (Postgres+ноды) обязательно, остальное по экономике. Код к этому уже подготовлен (#6).
