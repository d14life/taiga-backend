## Taiga — Current System Architecture (as-is)

Taiga ("Тайга") is a multi-provider uncensored-chat product. It is effectively **two monoliths plus a ring of half-extracted satellite modules**:

- **Backend monolith** — `mostik-ai/server.py`, ~14,729 lines, stdlib-only Python `ThreadingHTTPServer` (binds `127.0.0.1:8777`). One file holds the HTTP router, every `/api/*` handler, model routing, billing, storage, security, MCP, memory/RAG, media studios, routines, and the chat engine.
- **Frontend monolith** — `taiga-web/src/components/chat.tsx`, ~7,800 lines, whose `Chat()` god-component spans lines 451–6977 (165–189 `useState`, 27–28 `useEffect`). It is the composition root that mounts ~50 panels and prop-drills all state. Its transport engine `lib/use-taiga-chat.ts` (~1,327 lines) is the one *clean* layer on this side.

### What is tangled (cross-cutting coupling)

- **`chat()` is the gravity well.** L13158–14119 (~965 lines) inlines: rate/size guard, mode dispatch, auto-compact, auto-Brain decision, model resolution, RAG+memory+grounding+skills+MCP+UI-state system assembly, the tool-loop with `_emit_prose/_flush_prose` holdback closures, the `max_spend` cost-degradation ladder, Brain leader→expert orchestration, silent provider fallback on 3 exception/EOF paths, per-tool permission gating, and 3 billing branches. Billing correctness depends on hand-tuned byte-offset slicing (`len(buf)+len(hold)`). Brain is welded in via nonlocals (`out_total`, `expert_model`, `expert_usage`, `stream_model`) reassigned 800 lines apart. **Everything must be extracted around this; it comes last.**
- **Global mutable catalog state** (`CATALOG`, `RICH`, `PRICE`, `MODEL_KIND`, `DEFAULTS`, `CHEAP_MODEL`) is `.clear()`-then-refilled by a background refresher while request threads read it — the refill window is non-atomic.
- **Single shared SQLite connection + one `_DB_LOCK` RLock** serializes all writes; far-flung call sites grab `_DB_LOCK` by name for read-modify-write. Storage and the FTS search index are interleaved in the same functions (`chat_save` writes DB **and** indexes FTS inline).
- **"Extracted" satellite modules still reach back into the monolith.** `ad_gen.py`, `screen_copilot.py`, `video_rag.py` call `sys.modules[__name__]` to grab `server.venice_complete` etc. `agent_os` worker_runner closes over ~15 server globals. So the satellites are physically separate but logically still part of `server.py`.
- **Frontend mirror coupling**: `chat.tsx` ↔ `use-taiga-chat.ts`, plus `run-to-chat.ts`, `derive-goal.ts`, `run-loop-from-chat.ts` exist *solely* to bridge other features into the chat god-component. A Python caps schema (`skill_caps.py`) is hand-mirrored by `taiga-web/.../skill-caps.ts` with no drift guard.

### What is half-wired

- **Event-triggered routines are DEAD.** `POST /api/routine_event` (`api_routine_event` L11169, `_routine_fire_event` L14375) has **zero producers** anywhere in the repo (verified by grep — only the handler, the route, and comments exist). The frontend lets users create `on_run_done`/`on_chat_match` routines that save and display but can never fire. ~70 lines of infrastructure for a feature with no emitter.
- **Non-owner routines silently never run** (background engine is owner-only unless `MOSTIK_ROUTINES_ALL=1`); any user can save a routine that is inert with no feedback.
- **Two model catalogs** (Venice-only `CATALOG` vs multi-provider `RICH`) with three overlapping metadata sources (`CATALOG`→`OR_CTX`→`OR_LIVE`).
- **`beam` mode** is mechanically identical to `council` (always fuses) — `beam=True` only swaps a UI label and config namespace; the frontend `beamOn` flag is permanently `false`.
- **MCP OAuth** is a scaffold only (manual token paste); MCP resources/prompts are listed but there is no endpoint to read a resource or invoke a prompt.
- **`api_topup`** credits the wallet with **no payment step**, gated to HTTP 503 for non-owners; the only working credit paths are owner-manual `/api/billing` or flipping `test_topup`.
- **Internal sleep-time maintenance** (`memory_consolidate` at 04:00) is OFF by default behind `MOSTIK_SLEEPTIME=1`, so it never runs in the live deployment.

### What is dead

- **OpenRouter remnants** (`OR_MODELS`, `OR_CTX`, `OR_LIVE`, `_or_balance`, the `openrouter` branch in `headers_for`) — provider was removed (resale forbidden) but `OR_MODELS` (permanently `[]`) is still `.extend()`-ed into catalog payloads; no-ops that look like live wiring.
- **`RESALE_FORBIDDEN = set()`** — always empty, so the resale-gating branch in `resolve_key` is permanently dead despite `PROVIDER_LEGAL` metadata implying it should be enforced.
- **RAG legacy blob table** (`rag`) coexists with the new `rag_chunks` rows as a permanent dual-write fallback.
- **Frontend confirmed-dead** (~740 lines): `components/window-rearrange.tsx`, `components/resizable-shell.tsx`, `lib/features.ts`, `lib/session.ts` — zero imports (static or dynamic). (Note: ~15 panels that *look* dead are actually live via Next.js `dynamic()` in `settings-panel.tsx`.)
- **`orchestrator.py` raw `_complete` worker branch** (L184-210) — never exercised in prod because `worker_runner` is always injected.
- **`_read_script`** (skills_run.py), **`_VERDICT_REMARKS_RE`** (debate.py), **`_DB_MIGRATED`** flag (set, never read).

Two parallel scheduler systems that share no code: scheduler.py (owner orchestration cron, jobs.json) and the server.py routines block L14127-14480 (per-user LLM-prompt scheduler, routines.json), each with its own daemon thread, its own due-time calculator, and its own weekday parser — plus a third mini-scheduler (internal sleep-time) bolted into scheduler.py. Three implementations of 'fire a thing on a schedule'.
7. Global mutable catalog state mutated from request threads + a background refresher: load_rich_catalog does PRICE.clear()/MODEL_KIND.clear() then refills and rebinds RICH while request threads read them. The clear()/refill window is non-atomic — a concurrent read can see an empty PRICE. Routing/billing/chat are all tightly bound to these globals, making extraction hard.
8. save_memory never indexes FTS — a real correctness bug. save_memory writes only the JSON blob and never calls _fts_index_memory, so every remember/forget/extract/consolidate leaves the user-facing full-text search over remembered facts STALE until the next process restart. fts_sync exists and is wired for chats but there is no memory-write hook.
9. Dead event-trigger half of routines. POST /api/routine_event (L11169) and _routine_fire_event (L14375) have ZERO producers anywhere in the repo (grep-confirmed). The frontend lets users create on_run_done/on_chat_match routines that save, display, and can NEVER fire — a user-facing broken promise backed by ~70 lines of orphan infrastructure. Non-owner time-routines are similarly inert with no feedback.
10. 'Extracted' satellite modules still hard-bound to the monolith's global namespace via sys.modules[__name__] (ad_gen.py, screen_copilot.py, video_rag.py reach for server.venice_complete / server.VIDEO_MODELS / server.tool_fetch_url) and ~15-globals closures (agent_os worker_runner, the memory/RAG/grounding helpers). Plus pervasive duplication: the ~30 copy-pasted balance gates with inconsistent owner-exemption, the 4-5x media-handler preamble, the near-identical video/music SSE submit→poll→refund engines, the 35x load/save + 18x CustomEvent boilerplate on the frontend, and three separate tokenizer/stopword rankers in the context layer.

## Флаги безопасности (из аудита)
- CRITICAL — Forgeable owner auth: authorization is decided from the unauthenticated 'user' request-body field; is_owner('default') returns True (server.py L6971). Any client sending {"user":"default"} is owner and can set dev=true to run shell/run_code (arbitrary code execution) and read any file. Only api_users re-resolves the caller from a verified session token; the owner-gated admin routes and the dev/RCE chat path do NOT. Currently bounded only by the 127.0.0.1 bind — becomes full unauthenticated RCE + secret leak if proxied without external auth.
- HIGH — Unbounded owner file read: tool_read_file (L5269) / tool_list_dir (L5260) have NO path allowlist (unlike write/edit) and can read ~/.ssh/id_rsa, the provider key files, .cookie_key (the Fernet master key decrypting every BYOK key + cookie), and ~/.mostik/manifest-priv.key. tool_shell (L5277) runs subprocess.run(shell=True) with a 30s timeout but NO rlimits. Mitigated only by the owner gate — which is itself forgeable (above).
- HIGH — Frontend JS-skill RCE-in-origin: lib/skills/full-skills.ts:171 runs installed skill JS via new Function in the MAIN app window (full origin, localStorage, cookies, same-origin fetch); only console is shadowed. Skills are installable from GitHub/URL. Artifact runners are correctly sandboxed (iframe sandbox='allow-scripts' WITHOUT allow-same-origin) and Python skills go through Pyodide — only the JS skill path leaks into the host origin. Fix: route it through the sandboxed iframe.
- HIGH — MCP SSRF-via-redirect / TOCTOU: _mcp_rpc (L6791) and the notifications/initialized POST in _mcp_ensure (L6822) call raw urllib.request.urlopen WITHOUT the project's _ssrf_safe_opener(); _is_public_url is checked only on the initial URL at add/install time. The default opener auto-follows 3xx, so a public MCP server that 302-redirects to 169.254.169.254 (cloud metadata) or 127.0.0.1 has the redirect followed unguarded on every tool list/call. The codebase hardened other fetch paths against exactly this; the MCP RPC path was missed.
- HIGH — single Fernet key for everything: _cookie_fernet (L7576) generates ONE key at BASE/.cookie_key (plaintext, chmod 600) used for cookies, opt-in user files, AND every user's BYOK provider keys. If BASE is exfiltrated, the key travels with the ciphertext; one key compromises cookies + files + all users' provider keys at once. No rotation, no per-user keying.
- MEDIUM — Unauthenticated agent permit/answer injection: api_agent_permit (L12090) and api_agent_answer (L12106) authenticate nothing and key purely on a client-supplied run_id + tool_id into a process-global dict. A client that knows/guesses another run's run_id could inject an 'always'/'allow' decision into a different user's streamed agent run, defeating the interactive permission gate. No per-user binding.
- MEDIUM — Destructive-command guard is a regex denylist on the raw command string (_safety_hook L5793) — trivially bypassable (variable expansion, base64|sh, python os.system, find -delete). run_code rlimits cap CPU/RAM/procs but do NOT sandbox the filesystem or network (a tempdir cwd is not isolation). Defense-in-depth only, not a real boundary.
- MEDIUM — MCP token header pass-through: _mcp_headers (L6769) auto-prefixes 'Bearer ' only for Authorization; for a custom token_header the raw token is sent, and catalog/install headers are merged verbatim into outbound requests with no allowlist.
- MEDIUM — Orchestrator verify spoofing + unbounded BYOK body: the verify fallback heuristic (L5113-5117) flags 'true'/'да'/'прош' anywhere in the judge text as verified=true (a result echoing those words can spoof acceptance — verified is shown to the user as a trust signal); orchestrator _complete reads the raw response stream with no size cap, so a hostile BYOK base can return an unbounded body. Prompt-injection of the synth/verify steps via seed + prior results is acknowledged-as-text-only but not addressed.
- MEDIUM — Cinema-export as a free transcode/DoS surface: api_cinema_export downloads external scene URLs (SSRF-gated, good) then runs them through ffmpeg behind only rate-limit+balance; 60 scenes x 300s ffmpeg is heavy compute exposed to users, and the whole MP4 is returned base64 in JSON (memory blow-up).
- LOW — _is_public_url DNS-rebinding/TOCTOU: getaddrinfo validates here but urlopen re-resolves independently, so a host returning a public IP on first lookup and 127.0.0.1/169.254.169.254 on the second can slip past (requires attacker-controlled DNS + the fetch tools).
- LOW — ReDoS in on_chat_match routine pattern: user-supplied regex is run via re.search with only a 400-char cap and no backtracking/timeout guard; currently unreachable because the event path is dead, but becomes live the moment routine_event is wired.
- LOW — CSAM guard is a coarse English+Russian keyword AND-intersection (high false-negative via obfuscation/other languages); adequate as an upstream-key backstop, not a real safety control.
- LOW — Identity/provider-leak defense is prompt-only (taiga_identity + scrub_identity regex); scrub_identity does NOT run on the main streamed chat output, so a jailbroken uncensored model can leak the real underlying provider/model name to the user. Best-effort, not a hard boundary.
- LOW — Shared pool provider keys live as plaintext one-per-line files in $HOME, read on every request with no caching (only BYOK user keys are Fernet-encrypted).
