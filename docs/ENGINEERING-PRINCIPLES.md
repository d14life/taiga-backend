# Инженерные принципы — frontier-grade (как топ-инженеры в топ-компаниях)

КАЖДЫЙ бит системы должен соответствовать этому своду. Критик-совет проверяет по нему.
Не «работает» — а **безупречно, как сделал бы staff-инженер в Google/Meta/Stripe**.

## 1. Простота прежде всего (YAGNI + ponytail)
- Лестница перед кодом: нужно ли вообще? → stdlib/native? → есть зависимость? → 1 строка? → потом код.
- Минимум кода под задачу. НИКАКИХ преждевременных абстракций, спекулятивной гибкости, «на будущее».
- Удаляй мёртвое сразу. Меньше кода = меньше багов.

## 2. Структура и размер
- Один модуль/функция = ОДНА ответственность (SRP). Файл вырос — он делает слишком много, дроби.
- Маленькие, сфокусированные единицы с чётким интерфейсом; понятны без чтения внутренностей.
- DRY, но не в ущерб ясности. Дублирование лучше неправильной абстракции.
- Цель: chat.tsx (8157) и server.py (15125) → декомпозировать на сфокусированные модули.

## 3. Типы и контракты (строго)
- TS: strict, БЕЗ `any`, БЕЗ `@ts-ignore` без причины. Исчерпывающие union/switch (`never`-чек).
- Чёткие типы на границах (props, API, БД). Python: типы на публичных функциях, `ast`-валид.
- Контракт API = единый источник; фронт и бэк согласованы (см. BACKEND-API.md).

## 4. Обработка ВСЕХ состояний (не только happy-path)
- UI: loading (скелетоны под форму) · empty (красиво + как заполнить) · error (инлайн/тост) · success.
- Никогда не глотать ошибки. Падать ЧЕСТНО и громко, с понятным сообщением. Никаких пустых catch.
- Edge-кейсы: пусто, много, очень длинно, оффлайн, гонки, повтор, отмена.

## 5. Безопасность (по умолчанию)
- Валидация входа на границе. Authz: резолвить вызывающего из токена, не доверять телу (мина #0).
- Секреты — НЕ в коде/репо (env/файлы). Least privilege. Песочница + денилист для shell/run_code.
- SSRF/инъекции/CORS — закрыто (sec-коммиты 1-6). Параметризованный SQL, без конкатенации.

## 6. Производительность (осознанно)
- React: не лить useState на непрерывные значения (скролл/мышь) — useMotionValue. Мемоизация где нужно.
- Анимации только transform/opacity (GPU). backdrop-blur только на fixed/sticky.
- Бэк: без N+1, индексы, пагинация, стрим (SSE) для долгого. Ленивая загрузка тяжёлых панелей.
- Измеряй прежде чем оптимизировать; не угадывай.

## 7. Доступность (a11y, WCAG AA)
- Семантический HTML, клавиатурная навигация, фокус-видимость, ARIA где надо, контраст ≥ AA.
- Кнопка-текст читаем на фоне; reduced-motion уважать.

## 8. Читаемость и стиль
- Код читается как окружающий — те же идиомы, нейминг, плотность комментариев.
- Имена говорят намерение. Комментарии объясняют ПОЧЕМУ, не ЧТО. Самодокументируемо.
- Один стиль на проект: дизайн-токены, единый радиус/акцент/spacing. Без emoji, line/SVG иконки.

## 9. Тесты и верификация (доказательства, не вера)
- TDD где уместно: тест → провал → код → зелено. Юнит + интеграция + edge.
- Перед «готово»: tsc --noEmit=0, /app=200, смоук реально дёрнут, скриншот. Пруфы всегда.
- Поведение фич байт-в-байт по FEATURE-LEDGER. Регрессий ноль.

## 10. Ревью на каждый бит (критик-совет — ВСЕГДА в петле)
- КАЖДАЯ фича в петле проходит: внутренний критик-агент (Claude, multi-lens) + внешний СОВЕТ
  (`tools/external-critique.py --synth-only` LITE на каждый бит; `--deep` на сложное/важное).
- Пока совет не сказал ИДЕАЛЬНО — не «готово». Состязательно: «почему это НЕ идеально?».
- Никакого merge на красном. Один вынос = один коммит, никогда сломанное.

## 11. Прод-готовность (никаких полуфабрикатов)
- НЕТ placeholder/TODO-заглушек/«позже». Полная реализация (`full-output-enforcement`).
- Идемпотентность там, где важно (платежи, зачисления). Логи осмысленные, без console-шума.
- Конфиг через env, фичефлаги чисто. Обратная совместимость данных (миграции).
