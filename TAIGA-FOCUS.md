## 🧩 АРХИТЕКТУРНЫЙ ПРИНЦИП (Damir, 12.06) — «супер-фича vs коннектор»
Главное правило против раздувания продукта:

**1. Сами строим ТОЛЬКО «супер-фичи»** = комбинации многих методов, которых не даёт ни один
готовый MCP. Это наш ров: супер-поиск (Sonar+Exa+Brave+Venice вместе), оркестратор агентов,
co-browse, связка/мозг/совет, reels-генератор. «Всё предустановлено, методы работают вместе.»

**2. Всё, для чего УЖЕ есть MCP — НЕ строим нативно. Даём подключить коннектор.**
ComfyUI, Figma, Notion, Slack, GitHub, Supabase, Gmail и т.п. — это **MCP-коннектор**, а не наша
фича. Не переизобретаем, не раздуваем.

**3. В конструкторе Навыка / Команды / Агента — секция «Коннекторы (MCP)»** (как на скрине Claude
Routines: «Integrations available… Add connector»). Юзер вешает нужный MCP на свой агент/скилл.
Бэкенд MCP-клиент у нас УЖЕ есть (server.py: load_mcp_servers / mcp_list_tools / /api/mcp) —
осталось дать выбор коннектора в билдере (CORE UI + связать с конфигом агента).

**Тест, что строить нативно:** «есть ли это готовым MCP?» Да → коннектор. Нет, и это комбо методов →
супер-фича (строим). Это и есть depth-not-breadth применённый к интеграциям.
